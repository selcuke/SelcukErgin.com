I scanned the live site. It resolves and identifies itself as **“My Google AI Studio App”**, but the page content available to the crawler is essentially just the app shell, not rendered content. That pattern usually means **the HTML loads, then the frontend JavaScript fails before render**—which matches a Safari-only blank page symptom much more than a DNS or hosting issue. ([selcukergin.com][1])

The strongest likely cause is this:

**Your site is probably built with a modern Vite-style frontend target that Safari on some devices does not support.** Current Vite production builds default to a modern browser baseline, and Vite 7’s default browser target includes only **Safari 16.4+** unless you lower the target or add legacy support. Vite’s own docs also say older browsers need explicit legacy handling via `@vitejs/plugin-legacy`. ([vitejs][2])

That means if your Safari version is older, or if WebKit on that device chokes on a modern bundle feature, you get the classic result: **white/blank page, no visible server error**.

There are a few secondary possibilities too:

* a full-screen loading overlay never gets dismissed in Safari,
* a hydration/runtime crash from an unsupported API,
* a bad asset/module load after deployment,
* or a Replit publish/build mismatch. Replit’s own troubleshooting says to verify build/run commands and check deployment logs when a published app behaves differently from preview. ([Replit Docs][3])

## What I think is happening

Most likely order:

1. HTML is served correctly.
2. Safari downloads the JS bundle.
3. The JS bundle or one dependency crashes on Safari.
4. Since the app is client-rendered, you see a blank page.

C) Search the codebase for Safari-breaking patterns and fix them:
- top-level await in browser code
- unsupported modern syntax or APIs

- any fullscreen loader, animation, or overlay that may never resolve in Safari
- any code relying on window, document, localStorage, matchMedia, ResizeObserver, IntersectionObserver, or WebGL without guards

D) Add runtime protection:
- wrap app bootstrap in try/catch

- log failed asset/module loads

E) Check deployment-specific issues:
- verify the production build is serving the correct JS/CSS asset paths
- verify MIME types for JS modules are correct
- verify no 404s on hashed assets
- verify no base path issue
- verify Replit deployment serves the built frontend, not source files

F) Make these concrete code changes:
- update vite.config.* if present
- install and configure @vitejs/plugin-legacy if applicable
- fix any incompatible code found
- add an ErrorBoundary or equivalent app-level fallback
- ensure loading state always clears even if one async call fails

G) After changes:
- explain exactly what was wrong
- show the files changed
- give me the final diff summary
- tell me how to test Safari specifically

## Manual checks you should do in Safari

Use Safari Web Inspector on Mac/iPhone. WebKit’s official docs show how to enable remote inspection for iOS under **Settings → Safari → Advanced → Web Inspector**. ([WebKit][4])

In the Safari console, look for errors like:

* `SyntaxError`
* `Unexpected token`
* `Can't find variable`
* `TypeError: undefined is not an object`
* `Importing a module script failed`
* asset 404s for hashed JS/CSS files
